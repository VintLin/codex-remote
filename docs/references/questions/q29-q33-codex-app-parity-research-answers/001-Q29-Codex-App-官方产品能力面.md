## 不应进入近期 stage 的能力

近期 stage 应围绕“多设备 remote workbench 的最小闭环”：设备/项目/线程列表、start/continue/follow-up/steer/interrupt、实时 timeline、approval inbox、diff/review、命令输出、account/host health。

不应进入近期 stage 的能力：

1. **任意远程 shell / PTY / `process/*`**。官方协议明确 `process/*` 是 experimental、显式本地进程控制，并且运行在 Codex sandbox 外；近期只展示 Codex turn 内命令输出，最多做预定义 safe actions。([OpenAI Developers][2])
2. **自动批准、dangerFullAccess、session-wide approval 默认入口**。approval 是安全边界，不是便利按钮；近期只做窄授权、decline/cancel、明确 command/network/file 上下文。([OpenAI Developers][1])
3. **Git push / create PR / commit 作为默认主路径**。官方 App 有这些能力，但远程多设备误操作成本更高；先做 diff/review/read-only，再逐步引入 stage/revert/commit/push。([OpenAI Developers][1])
4. **Computer Use、Chrome extension、Browser Use 操作入口**。官方存在，但涉及屏幕、浏览器、登录态、剪贴板、GUI 前台控制；近期只消费 screenshots/browser artifacts。([OpenAI Developers][12])
5. **Plugins/apps/connectors 的安装、OAuth、Gmail/Drive/Slack 等副作用操作**。近期只暴露可用性与 enabled 状态；connector tool-call approvals 和破坏性工具审批需要单独安全设计。([OpenAI Developers][9]) ([OpenAI Developers][2])
6. **Automations / scheduled recurring work**。它不是当前远程工作台的最小闭环，会引入离线 host、重试、权限、幂等和任务审计问题。([OpenAI Developers][1])
7. **Realtime voice**。官方证据支持的是 voice dictation，不支持把它列为 Codex App 当前稳定“实时语音控制”能力。([OpenAI Developers][1])
8. **远程触发 Windows elevated sandbox setup**。可以展示状态和本地操作指引；不要让 Web 对远程 Windows host 做无本地确认的提权安装。([OpenAI Developers][2])
9. **Control Plane 保存 OpenAI/ChatGPT/provider secrets**。官方 remote 模型也强调 host 保留 project files、shell、app server、local context；Codex Remote 应保持 Worker 为 secrets/local FS/shell 的唯一持有者。([OpenAI Help Center][7])

## 结论

Codex Remote 的近期 App-like parity 不是“复刻 Codex App 所有 UI”，而是复刻它的**核心工作闭环**：项目/线程 → turn 输入 → 实时 timeline → approval → 文件/diff/命令输出 → review/follow-up。官方 remote control 资料已经把多 host、远程查看 live context、approval、terminal output、diffs、test results 明确为远程产品能力，因此这部分应作为 Codex Remote 的主线。([OpenAI Developers][3])

你的边界规则是正确的：公开 API 只暴露 Control Plane-shaped 产品语义；Worker 才能触碰 app-server、本机文件系统、Shell、Git、MCP、skills、account。这样可以把官方 Codex App 的本机能力投影成远程可控能力，而不是把 JSON-RPC 协议直接泄露成 Web API。
