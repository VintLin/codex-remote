# 参考项目架构调研报告 v0.2

## 结论摘要

本项目第一版应采用：

```text
Web UI
  -> Control Plane
    -> Device Worker
      -> localhost / unix socket Codex app-server
```

关键结论：

- Codex app-server 协议必须以 `openai-codex` 生成 schema 为事实源，本项目不要手写平行 Codex DTO。
- Worker 是唯一跨设备桥接层；Codex app-server 只绑定 loopback 或本机 socket。
- Control Plane 管 device registry、heartbeat、task board、conversation links 和审计，不保存 OpenAI / ChatGPT / provider secrets。
- Web MVP 的主轴是 `Device -> Project -> Conversation / Task`，不是单一聊天窗口，也不是 provider 管理器。
- iOS 后续扩展需要预留 pairing、trusted device、client heartbeat、Swift 类型生成，但第一版不实现 iOS App。
- `awslabs-cli-agent-orchestrator` 和 `Haleclipse-CodexDesktop-Rebuild` 不进入 MVP 主线，只保留远期参考价值。

已对 `project_referenecs` 下 7 个参考项目执行 `codegraph init` 并确认索引最新。其中 `openai-codex` 索引状态为 3,004 files、95,872 nodes、376,796 edges。

## 1. Codex App-server 协议面

### 结论

`packages/codex-protocol` 应直接从 Codex app-server schema 生成或拷贝上游生成产物并标记版本，不应在 Web、Worker、Control Plane 中重复定义 `Thread`、`Turn`、approval、notification 等结构。

### 代码证据

- `openai-codex/codex-rs/app-server-protocol/src/lib.rs:7-14` 导出 `generate_json`、`generate_ts`、`generate_types` 等 schema / 类型生成入口。
- `openai-codex/codex-rs/app-server-protocol/src/lib.rs:42` 统一导出 `protocol::v2::*`。
- `openai-codex/codex-rs/app-server-protocol/src/protocol/v2/thread_data.rs:132-180` 定义 `Thread`，包含 `id`、`session_id`、`model_provider`、`status`、`cwd`、`source`、`git_info`、`name`、`turns`。
- `openai-codex/codex-rs/app-server-protocol/src/protocol/v2/thread_data.rs:182-204` 定义 `Turn`，包含 `items`、`items_view`、`status`、`error`、时间戳和 duration。
- `openai-codex/codex-rs/app-server/src/request_processors/thread_processor.rs:162-184` 对 `thread/list` 的 `cwd` filter 做绝对路径归一化，支持单路径或多路径。
- `openai-codex/codex-rs/app-server/src/request_processors/thread_processor.rs:2135-2188` 显示 `thread/read` 会按 loaded / persisted 状态合成视图，`includeTurns` 时行为不同。
- `openai-codex/codex-rs/app-server/src/in_process.rs:20-31` 明确 in-process 路径仍返回 JSON-RPC envelope，并且 server requests 不应静默丢失。

### 采用

- 新增 `packages/codex-protocol`，保存当前 Codex 版本的 generated TS / JSON Schema。
- Worker 内部用 app-server 原始类型，但 Worker 对 Control Plane 暴露稳定领域模型。
- `thread/list` 的 `cwd` filter 作为项目维度检索首选路径。
- `thread/read(includeTurns=true)` 作为只读详情主路径，必要时 `thread/resume` 后再操作。

### 暂缓

- 不在 v0.1 开启 experimental API 作为必需能力。
- 不读取本地 SQLite / rollout 作为写接口，只作为诊断 fallback。

### 不采纳

- 不在 API handler / UI / tests 中手写一份平行 Codex Thread/Turn DTO。
- 不把 Codex app-server 全量类型直接外泄为 Control Plane 公共 API。

## 2. Worker 与本机 App-server 边界

### 结论

Worker 负责连接、启动、探测和保护本机 Codex app-server。app-server 是本机进程间协议，不是跨设备协议。

### 代码证据

- `openai-codex/codex-rs/app-server-transport/src/transport/mod.rs:72-78` 定义 `Stdio`、`UnixSocket`、`WebSocket`、`Off` 四种 transport。
- `openai-codex/codex-rs/app-server-transport/src/transport/mod.rs:111-150` 解析 `stdio://`、`unix://`、`ws://IP:PORT` 和 `off`。
- `openai-codex/codex-rs/app-server-transport/src/transport/websocket.rs:70-76` 启动 banner 明确 loopback 可用于 localhost / SSH forwarding，非 localhost 需要 WebSocket auth。
- `openai-codex/codex-rs/app-server-transport/src/transport/websocket.rs:89-103` 拒绝带 `Origin` header 的请求，降低浏览器跨站误连风险。
- `openai-codex/codex-rs/app-server-transport/src/transport/websocket.rs:129-142` 非 loopback 且未配置 auth 时拒绝启动。
- `openai-codex/codex-rs/app-server-transport/src/transport/auth.rs:27-56` 暴露 `--ws-auth`、token file、token sha256、signed bearer secret 等认证参数。
- `openai-codex/codex-rs/app-server-transport/src/transport/auth.rs:69-85` 支持 capability token 和 signed bearer token 两类配置。

### 参考实现证据

- `Sunwood-ai-labs-codex-remote-control-lab/scripts/start-phone.js:66-89` 将 Codex URL 默认指向 `ws://127.0.0.1:<port>`，UI server 可监听 `0.0.0.0`，但 app-server 仍在本机侧。
- `Sunwood-ai-labs-codex-remote-control-lab/scripts/start-phone.js:482-568` `AppServerRpcClient` 维护 upstream WebSocket、pending map、初始化状态和断线 reset。
- `Sunwood-ai-labs-codex-remote-control-lab/scripts/start-phone.js:507-514` 连接打开后先发 `initialize`，成功后发 `initialized`。
- `Sunwood-ai-labs-codex-remote-control-lab/scripts/start-phone.js:570-589` 可由桥接层启动 `codex app-server --listen <url>`。
- `Sunwood-ai-labs-codex-remote-control-lab/scripts/start-phone.js:731-735` 对外请求通过 URL token gate。
- `Sunwood-ai-labs-codex-remote-control-lab/scripts/start-phone.js:738-752` 对路径访问做 base path 限制。

### 采用

- `apps/worker` 只连接本机 app-server；默认使用 loopback WebSocket，后续补 Unix socket。
- Worker 可负责启动 app-server，但需要记录 `startedByWorker` 和进程状态。
- Worker 必须实现 reconnect、pending request timeout、overload retry、JSON-RPC envelope 校验。
- Worker 对外只暴露自己的 API，不裸露 app-server URL。

### 暂缓

- Unix socket 先作为 macOS / Linux 优化项，第一阶段 probe 先覆盖 loopback WebSocket。
- signed bearer token 可作为后续 Worker->app-server auth 增强，MVP 先让 app-server 保持本机监听。

### 不采纳

- 不把 Codex app-server 直接绑定公网或 LAN 作为产品能力。
- 不把 URL query token 作为长期身份模型；它只能作为 demo 级参考。

## 3. Control Plane、设备注册与多端连接

### 结论

Control Plane 应拥有自己的设备与任务模型，不应把 app-server thread 当作系统唯一实体。设备、项目、conversation、task、conversation link 是不同层级。

### 代码证据

- `getpaseo-paseo/packages/protocol/src/daemon-endpoints.ts:15-19` 将 relay role、relay protocol version、默认 relay endpoint 明确建模。
- `getpaseo-paseo/packages/protocol/src/daemon-endpoints.ts:52-82` 独立解析 host/port/IPv6，避免连接地址散落在业务逻辑中。
- `getpaseo-paseo/packages/protocol/src/daemon-endpoints.ts:169-180` 用 `buildRelayWebSocketUrl` 生成带 `serverId`、role、version 的 relay URL。
- `getpaseo-paseo/packages/server/src/server/server-id.ts:23-34` 生成并持久化稳定 `serverId`。
- `getpaseo-paseo/packages/client/src/daemon-client.ts:853-897` `DaemonClient` 持有 transport、listener、reconnect、connection state、runtime metrics 等连接状态。
- `getpaseo-paseo/packages/client/src/daemon-client.ts:899-915` 客户端区分 direct / relay，并记录 serverId / clientId。
- `getpaseo-paseo/packages/client/src/daemon-client.ts:1565-1578` client heartbeat 包含 `deviceType`、`focusedAgentId`、`lastActivityAt`、`appVisible`。
- `getpaseo-paseo/packages/server/src/server/session.ts:4472-4491` server 端处理 client heartbeat 并保存 activity。

### 采用

- Control Plane DB 至少需要 `devices`、`remote_projects`、`codex_conversations`、`board_projects`、`board_tasks`、`conversation_links`、`audit_logs`。
- Device 需要稳定 `deviceId`，并预留 `serverId`、`connectionPath: direct | relay`、`lastSeenAt`。
- Worker heartbeat 需要包含 worker version、Codex version、OS、connection status、当前 running turn 数。
- Task 与 ConversationLink 是 Control Plane 自有实体，不能塞回 Codex thread metadata。

### 暂缓

- 公共 relay 不做 MVP，但 API contract 预留 connectionPath 和 pairing 字段。
- 多用户 RBAC 暂缓，但 audit log 保留 actor 字段。

### 不采纳

- 不用 Codex `threadId` 充当全局 task id。
- 不做自动任务迁移或自动设备调度。

## 4. 不依赖同账号的安全边界

### 结论

本项目要绕开官方同账号 remote control 假设：Control Plane 只信任 Worker 身份，不处理 OpenAI 账号身份。每台设备的 Codex auth/API key/model provider 留在本机。

### 代码证据

- `openai-codex/codex-rs/app-server-transport/src/transport/auth.rs:27-56` 说明 app-server WebSocket auth 是本地 transport auth，不是跨设备产品身份。
- `Sunwood-ai-labs-codex-remote-control-lab/scripts/start-phone.js:83-89` bridge 可以对外监听并启用 token，而 Codex URL 仍指向本地。
- `friuns2-codex-mobile/src/server/authMiddleware.ts:8-11` 使用 Web UI session cookie 和本地 session store。
- `friuns2-codex-mobile/src/server/authMiddleware.ts:76-82` session store 放在 `CODEX_HOME` 下，而不是远端账号系统。
- `friuns2-codex-mobile/src/server/authMiddleware.ts:106-116` 持久化 session token 时用 `0o600` 文件权限。
- `Emanuele-web04-remodex/phodex-bridge/src/secure-device-state.js:19-56` 以本地文件 / Keychain 维护 bridge identity。
- `Emanuele-web04-remodex/phodex-bridge/src/secure-device-state.js:113-130` 保存 trusted phone public key。
- `Emanuele-web04-remodex/phodex-bridge/src/secure-device-state.js:177-190` bridge 生成 ed25519 identity key 和 device id。
- `Emanuele-web04-remodex/phodex-bridge/src/secure-transport.js:25-34` 定义 pairing QR version、secure protocol、handshake mode、pairing 过期时间、buffer 限制。
- `Emanuele-web04-remodex/phodex-bridge/src/secure-transport.js:59-69` pairing payload 包含 relay、sessionId、macDeviceId、macIdentityPublicKey、expiresAt。
- `Emanuele-web04-remodex/phodex-bridge/src/secure-transport.js:136-193` handshake 校验 protocolVersion、sessionId、phone identity、handshake mode、QR 过期和 trusted reconnect。
- `Emanuele-web04-remodex/phodex-bridge/src/qr.js:42-60` 打印 QR 时避免输出包含私有 relay metadata 的 pairing JSON。

### 采用

- Worker pairing / registration 使用本项目 device token 或 one-time pairing token。
- Control Plane 的 `Add Device` 生成短期 `PairingOffer`，Web 显示一条目标机器可执行的 Worker setup 命令。
- Worker setup 使用 pairing token 注册后换取长期 device token；pairing token 立即失效。
- Worker 主动出站连接 Control Plane，避免要求目标设备开放入站端口。
- 初次接入只完成设备注册和 Codex 探测；project allowlist 在 Web 中后置确认，设备在确认前处于 `needs setup`。
- Control Plane DB 不存 Codex auth、OpenAI key、provider key、ChatGPT session。
- Worker audit log 至少记录：actor、deviceId、projectId、conversationId、action、target、summary、timestamp。
- Project allowlist 是 Worker 本地安全边界，Control Plane 只下发/展示，不越权访问文件系统。

### 暂缓

- E2EE secure transport、trusted reconnect、短码配对放入 iOS 阶段。
- Keychain / Windows Credential Manager 等本机 secret store 封装放入 Worker 安装阶段。

### 不采纳

- 不复制官方 remote control 的同 ChatGPT 账号注册模型。
- 不把 provider proxy、OpenRouter key、custom endpoint key 做成 Control Plane 功能。
- 不要求用户手写 WebSocket URL、device token、app-server URL 等底层连接配置。
- 不在 MVP 要求用户安装系统服务或正式桌面安装包。

## 8. 对 Turborepo 包边界的影响

建议结构：

```text
apps/
  web/
  control-plane/
  worker/

packages/
  codex-protocol/
  api-contract/
  db/
  shared/
  ui/
```

包职责：

- `packages/codex-protocol`：上游 app-server generated TS / JSON Schema、版本元数据、Worker 内部 adapter。
- `packages/api-contract`：Control Plane <-> Web / Worker / future iOS 的唯一 API 契约。
- `packages/db`：Control Plane DB schema 和 repository，持久化字段事实源。
- `packages/shared`：不含 IO 的通用类型、日期、状态映射、错误码。
- `packages/ui`：Web 组件，不能依赖 Worker 或 Codex protocol。
- `apps/worker`：连接 app-server、本机项目/git/terminal、project allowlist、audit log。
- `apps/control-plane`：device registry、task board、conversation link、event fanout。
- `apps/web`：工作台 UI，只调用 Control Plane API。

依赖方向：

```text
apps/web -> packages/api-contract, packages/ui, packages/shared
apps/control-plane -> packages/api-contract, packages/db, packages/shared
apps/worker -> packages/api-contract, packages/codex-protocol, packages/shared
packages/api-contract -> packages/shared
packages/db -> packages/shared
packages/ui -> packages/api-contract, packages/shared
```

禁止方向：

- `apps/control-plane` 不依赖 `packages/codex-protocol`。
- `apps/web` 不依赖 `packages/codex-protocol`。
- `packages/codex-protocol` 不依赖 Control Plane / DB / UI。

## 9. 第一阶段 Worker Probe 任务清单

### 目标

证明单设备 Worker 能稳定控制本机 Codex app-server，并能输出可诊断 JSON summary。

### 任务

1. 初始化 pnpm workspace + Turborepo。
2. 创建 `packages/codex-protocol`，导入或生成当前 Codex app-server TS / JSON Schema，并记录 Codex version。
3. 创建 `apps/worker`。
4. 实现 `AppServerRpcClient`：
   - 支持 loopback WebSocket；
   - 发送 `initialize` 和 `initialized`；
   - request id、pending map、timeout；
   - notification callback；
   - disconnect reset；
   - JSON-RPC envelope 校验。
5. 实现 `worker probe` CLI：
   - `initialize`；
   - `model/list`；
   - `thread/list`，含 `cwd` filter；
   - `thread/read(includeTurns=true)`；
   - `thread/resume`；
   - `thread/start`；
   - `turn/start`；
   - streaming notifications；
   - `turn/steer`；
   - `turn/interrupt`；
   - server request / approval request 捕获，不要求自动批准。
6. 实现安全检查：
   - app-server URL 必须是 loopback 或本机 socket；
   - probe summary 不输出 secrets；
   - project path 必须在 allowlist 内；
   - 失败返回 error code、method、stage、message。
7. 在 macOS 跑 probe。
8. 在 Windows 跑 probe。
9. 把差异写回技术规格。
10. 实现 `worker setup/start/status/logs` CLI：
    - `setup --server <url> --pair <token> [--name <name>]`；
    - 生成并持久化本机 `deviceId`；
    - 用 pairing token claim，换取长期 device token；
    - 本地保存 server URL、deviceId、device name、app-server transport；
    - device token 优先进入本机 secret store，MVP fallback 为 `0600` 权限文件；
    - setup 完成后自动运行 Codex 探测并上报 `needs setup`。
11. 实现 Control Plane pairing：
    - `pairing.offer.create` 生成短期 token 和 install command；
    - `pairing.offer.claim` 一次性换取 device token；
    - claim 后 pairing token 立即失效；
    - Web `Add Device` 显示命令、过期时间和目标 server URL。
12. 实现 Web allowlist 确认：
    - 新设备先显示 `needs setup`；
    - 用户确认可访问项目目录后，Worker 才允许列项目和对话；
    - 审计记录 allowlist 变更。

## 10. 风险与决策

| 风险 | 影响 | 决策 |
| --- | --- | --- |
| app-server schema 变动 | Worker adapter 失效 | `codex-protocol` 版本化，probe 先验 schema |
| approval request 未处理 | 任务卡死 | Worker event stream 必须上报 server request |
| app-server 暴露范围过大 | 本机文件/执行能力外泄 | app-server 只 loopback/socket，跨设备只走 Worker |
| Control Plane 保存 secrets | 违背核心定位 | DB schema 禁止 secrets 字段，文档和测试 fixture 用 REDACTED |
| iOS 后续重构 | API 大改 | v0.1 contract 预留 pairing、trusted device、heartbeat |
| 范围膨胀到 provider proxy | MVP 失焦 | provider proxy 明确不采纳 |

## 11. 最终建议

从零到一第一步不是直接做完整 Web UI，而是：

1. 初始化 Turborepo 的最小骨架。
2. 建 `packages/codex-protocol` 和 `apps/worker`。
3. 完成 Worker probe。
4. 再建 Control Plane DB/API。
5. 最后做 Web MVP。

这样能先验证最脆弱的协议和安全边界，避免在 UI 完成后发现 app-server 行为或跨设备边界需要返工。
