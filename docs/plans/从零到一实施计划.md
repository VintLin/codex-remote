# 从零到一实施计划

## 目标

把项目从 PRD 推进到可实施状态。第一阶段不追求完整产品 UI，而是验证技术地基：

> 自托管 Control Plane 能否通过每台设备的 Worker，稳定控制各自本地配置下的 Codex app-server。

## 阶段 0：当前状态核对

目的：

确认当前未跟踪和已修改的工程文件是否属于可继续使用的项目骨架，避免把未验证的 scaffold 当作已完成基础设施。

任务：

1. 查看 `git status --short`，记录与文档整理无关的既有状态。
2. 记录当前已有 Web/UI 工作：`apps/web`、`packages/ui`、`pnpm-lock.yaml`、`scripts/`、UI 截图和新的 superpowers specs/plans。
3. 检查 `package.json`、`pnpm-workspace.yaml`、`turbo.json`、`apps/web/package.json`、`packages/ui/package.json` 是否与技术规格一致。
4. 检查 `apps/web/.next/`、`.turbo/`、`node_modules/` 是否应被 `.gitignore` 排除。
5. 运行已建立的验证命令；若命令尚未建立，在执行结果中明确记录。
6. 不在本阶段清理、回滚或提交与文档整理无关的文件。

验收：

- 当前工程骨架状态有明确记录。
- 文档整理提交只包含 canonical docs 和 archives 路径。
- 未验证的工程文件不会被实施计划标记为已完成。

## 阶段 1：文档与边界固化

产物：

- `docs/specs/多设备 Codex 控制台 技术规格.md`
- `docs/references/research/参考项目架构调研报告 v0.2.md`
- `docs/archives/plans/`
- `AGENTS.md`

验收：

- 明确不依赖同一 OpenAI / ChatGPT 账号。
- 明确 Control Plane 不保存 OpenAI secrets。
- 明确 Worker 是唯一对外桥接层。
- 明确 Codex app-server 不直接暴露到 LAN / 公网。
- 明确第一版是 Web，后续预留 iOS。
- 明确 Turborepo monorepo 结构和包依赖方向。

## 阶段 2：Turborepo 最小骨架核对

目的：

建立最小工程结构，不提前做 UI。

建议目录：

```text
apps/worker
packages/codex-protocol
packages/shared
```

任务：

1. 初始化 `package.json`、`pnpm-workspace.yaml`、`turbo.json`。
2. 建立 `apps/worker`。
3. 建立 `packages/codex-protocol`。
4. 建立 `packages/shared`。
5. 配置 `lint`、`typecheck`、`test`、`build` 占位脚本。

验收：

- `pnpm install` 成功。
- `pnpm typecheck` 可运行。
- `pnpm test` 可运行。
- `pnpm build` 可运行。

## 阶段 3：Codex Protocol 事实源

目的：

把 Codex app-server schema 固化为 Worker 内部协议事实源。

任务：

1. 生成或导入 app-server TypeScript schema。
2. 生成或导入 app-server JSON Schema。
3. 记录 Codex version。
4. 增加 schema version metadata。
5. 增加最小 adapter：`Thread -> WorkerConversationSnapshot`。

验收：

- `packages/codex-protocol` 不依赖 Web / Control Plane。
- 修改 generated schema 后，Worker adapter 能通过类型检查暴露不兼容点。
- 不存在手写平行 Codex Thread / Turn 类型。

## 阶段 4：Worker Probe

目的：

先验证 Codex app-server 是否满足 MVP 控制需求。

任务：

1. 实现 loopback WebSocket app-server client。
2. 支持 `initialize` / `initialized`。
3. 支持 JSON-RPC request id、pending map、timeout、disconnect reset。
4. 支持 notification stream。
5. 验证 `model/list`。
6. 验证 `thread/list`，含 `cwd` filter。
7. 验证 `thread/read(includeTurns=true)`。
8. 验证 `thread/resume`。
9. 验证 `thread/start`、`turn/start`。
10. 验证 streaming notifications。
11. 验证 `turn/steer`。
12. 验证 `turn/interrupt`。
13. 捕获 server request / approval request。
14. 输出 JSON summary。

验收：

- 单设备 probe 可稳定完成。
- 输出 JSON summary。
- 失败路径有 method、stage、message。
- 不上传 OpenAI / ChatGPT / provider secrets。
- app-server URL 必须是 loopback 或本机 socket。

## 阶段 5：最小 Worker

目的：

把 probe 变成可被 Control Plane 调用的 Worker。

任务：

1. Worker 提供本机 HTTP/WebSocket API。
2. Worker 只连接 localhost / socket app-server。
3. Worker 支持 device token。
4. Worker 支持 project allowlist。
5. Worker 支持 heartbeat。
6. Worker 输出 audit log。
7. Worker 把 app-server events 归一化为 Control Plane events。
8. Worker 提供 `setup`、`start`、`status`、`logs` CLI。
9. `setup` 支持 `--server`、`--pair`、`--name` 参数。
10. `setup` 生成本机 `deviceId`，用一次性 pairing token 注册并换取长期 device token。
11. Worker 本机保存 `WorkerLocalConfig`，但不把 device token 写入普通 JSON。
12. Worker 首次上线后状态为 `needs setup`，等待 Web 确认 project allowlist。

验收：

- Control Plane 可以注册一台 Worker。
- 用户可以复制 Web 生成的一条命令，在目标机器完成 Worker setup。
- pairing token 使用一次后失效。
- Worker 重启后可用本机保存的 device token 自动重连。
- Worker 可以返回项目列表、conversation 列表、conversation 内容。
- Worker 可以启动、follow-up、中止一个 Codex turn。
- Worker 可以上报 pending approval。

## 阶段 6：Control Plane

目的：

建立多设备状态聚合和任务看板的后端。

建议目录：

```text
apps/control-plane
packages/db
packages/api-contract
packages/shared
```

任务：

1. 建立 API contract。
2. 建立 DB schema。
3. 建立 device registry。
4. 建立 Worker connection/session 管理。
5. 建立 heartbeat ingest。
6. 建立 Board Project / Task / ConversationLink API。
7. 建立 event fanout 给 Web UI。
8. 建立 `pairing.offer.create` 和 `pairing.offer.claim`。
9. Add Device 时生成短期 pairing token 和安装命令。
10. claim 成功后签发长期 device token，并把 pairing token 置为失效。
11. 建立设备 `needs setup` 状态和 project allowlist 确认流程。

验收：

- 两台 Worker 可同时在线。
- 能展示每台设备状态和最后心跳。
- Web 可以生成目标机器可执行的 pairing command。
- 未确认 allowlist 的设备不能访问本机项目。
- 能保存 task 与 conversation link。
- 重启 Control Plane 后任务看板仍存在。
- Control Plane DB 不含 OpenAI / ChatGPT / provider secrets 字段。

## 阶段 7：Web MVP

目的：

做第一版网页控制台。

建议目录：

```text
apps/web
packages/ui
packages/api-contract
```

页面：

- 设备视图。
- Add Device / pairing command。
- 设备详情。
- Project allowlist 确认。
- 项目 / 对话列表。
- 对话输出流。
- pending approval 提示。
- 输入 follow-up。
- 中止按钮。
- 任务看板。
- Task 详情和关联 conversations。

验收：

- 两台设备在线状态显示正确。
- 能从 Web 创建 pairing offer，并复制目标机器接入命令。
- 目标机器完成 Worker setup 后，Web 显示设备 `needs setup`。
- 用户确认 project allowlist 后，设备进入可用状态。
- 能在 Web 中切换设备。
- 能打开指定设备上的 conversation。
- 能发送 follow-up。
- 能中止任务。
- 能把不同设备的 conversations 关联到同一 task。

## 阶段 8：iOS 扩展准备

第一版不做 iOS App，但 API 设计要预留：

- `PairingOffer`
- `TrustedDevice`
- `ClientHeartbeat`
- `DeviceConnectionStatus`
- OpenAPI / JSON Schema 生成入口

约束：

- mobile client 只连接 Control Plane 或 future relay。
- iOS 不直接连接 Worker 或 Codex app-server。
- iOS 不持有 OpenAI / Codex secrets。

## 第一阶段推荐任务顺序

1. 初始化 Turborepo。
2. 建立 `packages/codex-protocol`。
3. 导入 / 生成 app-server schema。
4. 建立 `apps/worker`。
5. 写 loopback app-server client。
6. 写 `worker probe`。
7. 在 MacBook 上跑 probe。
8. 在 Windows 设备上跑 probe。
9. 根据差异更新技术规格。
10. 实现 Worker `setup/start/status/logs` CLI。
11. 实现 Control Plane pairing offer / claim。
12. 实现 Add Device 页面和 pairing command。
13. 再进入 Control Plane 设备状态、任务看板和 Web MVP。
