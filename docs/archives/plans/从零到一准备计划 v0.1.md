# 从零到一准备计划 v0.1

## 目标

把项目从 PRD 推进到可实施状态。第一阶段不追求完整产品 UI，而是验证技术地基：

> 自托管 Control Plane 能否通过每台设备的 Worker，稳定控制各自本地配置下的 Codex app-server。

## 阶段 0：文档与边界固化

产物：

- `docs/specs/多设备 Codex 控制台 技术规格 v0.2.md`
- `docs/references/research/参考项目技术调研 v0.1.md`
- `docs/plans/参考项目架构调研计划 v0.1.md`
- `AGENTS.md`

验收：

- 明确不依赖同一 OpenAI / ChatGPT 账号。
- 明确 Control Plane 不保存 OpenAI secrets。
- 明确 Worker 是唯一对外桥接层。
- 明确第一版是 Web，后续预留 iOS。
- 明确 Turborepo monorepo 结构。

## 阶段 1：协议探针

目的：

先验证 Codex app-server 是否满足 MVP 控制需求。

建议目录：

```text
apps/worker
packages/codex-protocol
packages/shared
```

任务：

1. 初始化 pnpm + Turborepo。
2. 生成 app-server TypeScript schema 和 JSON Schema。
3. 写 `worker probe`，连接本机 Codex app-server。
4. 验证 `initialize`、`model/list`、`thread/list`、`thread/read`。
5. 验证 `thread/start`、`turn/start`、streaming events。
6. 验证 `turn/steer` 和 `turn/interrupt`。
7. 记录 macOS 和 Windows 差异。

验收：

- 单设备 probe 可稳定完成。
- 输出 JSON summary。
- 失败路径有明确错误原因。
- 不上传 OpenAI secrets。

## 阶段 2：最小 Worker

目的：

把 probe 变成可被 Control Plane 调用的 Worker。

任务：

1. Worker 提供本机 HTTP/WebSocket API。
2. Worker 只连接 localhost app-server。
3. Worker 支持 device token。
4. Worker 支持 project allowlist。
5. Worker 支持 heartbeat。
6. Worker 输出 audit log。

验收：

- Control Plane 可以注册一台 Worker。
- Worker 可以返回项目列表、conversation 列表、conversation 内容。
- Worker 可以启动、follow-up、中止一个 Codex turn。

## 阶段 3：Control Plane

目的：

建立多设备状态聚合和任务看板的后端。

建议目录：

```text
apps/control-plane
packages/db
packages/api-contract
packages/shared
```

任务：

1. 建立 DB schema。
2. 建立 device registry。
3. 建立 Worker connection/session 管理。
4. 建立 Board Project / Task / ConversationLink API。
5. 建立 event fanout 给 Web UI。

验收：

- 两台 Worker 可同时在线。
- 能展示每台设备状态和最后心跳。
- 能保存 task 与 conversation link。
- 重启 Control Plane 后任务看板仍存在。

## 阶段 4：Web MVP

目的：

做第一版网页控制台。

建议目录：

```text
apps/web
packages/ui
packages/api-contract
```

页面：

- 设备视图。
- 设备详情。
- 项目 / 对话列表。
- 对话输出流。
- 输入 follow-up。
- 中止按钮。
- 任务看板。
- Task 详情和关联 conversations。

验收：

- 两台设备在线状态显示正确。
- 能在 Web 中切换设备。
- 能打开指定设备上的 conversation。
- 能发送 follow-up。
- 能中止任务。
- 能把不同设备的 conversations 关联到同一 task。

## 阶段 5：iOS 扩展准备

第一版不做 iOS App，但 API 设计要预留：

- Control Plane API 不绑定浏览器 session。
- API contract 可生成 OpenAPI / JSON Schema。
- mobile client 只连接 Control Plane，不直接连接 Worker 或 Codex app-server。
- iOS 不持有 OpenAI / Codex secrets。
- 未来通过 QR code 绑定 Control Plane 账号或 device session。

## 第一阶段推荐任务顺序

1. 初始化 Turborepo。
2. 完成 codegraph-backed 参考项目架构调研。
3. 根据调研更新技术规格和计划。
4. 增加 `packages/codex-protocol`。
5. 生成 app-server schema。
6. 增加 `apps/worker`。
7. 写 `worker probe`。
8. 在 MacBook 上跑 probe。
9. 在 Windows 设备上跑 probe。
10. 根据差异更新技术规格。

## 暂不处理

- UI 视觉设计。
- iOS App。
- provider proxy。
- 多用户权限系统。
- 多 agent orchestration。
- CI/CD 发布。
