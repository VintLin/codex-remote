# 参考项目架构调研计划 v0.1

## 目标

本轮调研不是做 README 摘要，而是站在架构决策角度回答：

> 多设备 Codex 控制台第一版应该如何划分 Web、Control Plane、Worker、Codex app-server、本地配置与未来 iOS client 的边界。

调研结果只服务于从零到一准备，不扩散到 provider proxy、多 agent 编排、桌面重打包、移动端全量实现。

## 调研原则

- 先查协议事实源，再查参考实现。
- 每个结论必须能对应到代码位置、协议字段、进程边界或安全边界。
- 不用“项目看起来相似”作为采纳理由；必须说明能减少哪类实现风险。
- 不把参考项目的产品范围带入本项目 MVP。
- 对低相关项目只记录排除理由，不做等量深挖。

## 必答问题

### 1. Codex app-server 协议面

结论方向：

- `openai-codex` 的 `app-server-protocol/schema/typescript/v2` 是协议类型事实源。
- 本项目不手写平行 Codex DTO；`packages/codex-protocol` 从上游 schema 生成或复制并标记版本。
- Worker 对 Control Plane 暴露自己的稳定领域模型，不能直接把全部 app-server 类型外泄。

重点代码：

- `openai-codex/codex-rs/app-server-protocol/src/lib.rs`
- `openai-codex/codex-rs/app-server-protocol/src/protocol/v2/thread_data.rs`
- `openai-codex/codex-rs/app-server-protocol/schema/typescript/v2/*`
- `openai-codex/codex-rs/app-server/src/request_processors/thread_processor.rs`

需要确认：

- `initialize`、`model/list`、`thread/list`、`thread/read`、`thread/start`、`turn/start`、`turn/interrupt` 的最小参数。
- `thread/list` 的 `cwd` filter 是否足以做项目维度检索。
- `thread/read includeTurns` 对已加载 / 未加载线程的行为差异。
- server requests 和 approval responses 的完整生命周期。

### 2. Worker 与 app-server 边界

结论方向：

- app-server 应只监听 localhost / unix socket。
- Worker 是唯一跨设备暴露的桥接层。
- 第一版 Worker 先做 request/response + event fanout，不做完整 Codex Desktop 克隆。

重点代码：

- `openai-codex/codex-rs/app-server-transport/src/transport/mod.rs`
- `openai-codex/codex-rs/app-server-transport/src/transport/websocket.rs`
- `openai-codex/codex-rs/app-server-transport/src/transport/auth.rs`
- `Sunwood-ai-labs-codex-remote-control-lab/scripts/start-phone.js`
- `Sunwood-ai-labs-codex-remote-control-lab/scripts/history-sync.js`
- `Sunwood-ai-labs-codex-remote-control-lab/scripts/thread-read.js`

需要确认：

- localhost WebSocket 与 unix socket 在 macOS / Windows 的差异。
- Worker 是否负责启动 app-server，还是只连接已存在 app-server。
- app-server 断线、重启、schema 变化时 Worker 如何恢复。

### 3. 多设备 Control Plane

结论方向：

- Control Plane 管 device registry、task board、conversation links、event fanout。
- Control Plane 不保存 OpenAI / ChatGPT / provider secrets。
- 每台设备保留自己的 Codex auth、API key、model provider、project allowlist。

重点代码：

- `getpaseo-paseo/packages/protocol/src/daemon-endpoints.ts`
- `getpaseo-paseo/packages/client/src/daemon-client.ts`
- `getpaseo-paseo/packages/server/src/server/session.ts`
- `getpaseo-paseo/packages/server/src/server/relay-transport.ts`

需要确认：

- deviceId / serverId / clientId 的命名和生命周期。
- heartbeat、visibility、focused task 的事件模型。
- direct connection 与 future relay 的抽象是否要在 v0.1 数据模型预留。

### 4. 安全与配对

结论方向：

- 第一版 Web 可用设备 token + project allowlist + audit log。
- 后续 iOS 需要 QR pairing / trusted device / short-lived session。
- 不继承官方 remote control 的同账号 ChatGPT auth 假设。

重点代码：

- `openai-codex/codex-rs/app-server-transport/src/transport/auth.rs`
- `getpaseo-paseo/packages/server/src/server/pairing-offer.ts`
- `getpaseo-paseo/packages/server/src/server/pairing-qr.ts`
- `Emanuele-web04-remodex/phodex-bridge/src/secure-device-state.js`
- `Emanuele-web04-remodex/phodex-bridge/src/secure-transport.js`
- `Emanuele-web04-remodex/phodex-bridge/src/qr.js`

需要确认：

- device token 存储位置和轮换方式。
- pairing session 过期策略。
- trusted device 是否一对多，是否允许多手机 / 多浏览器。
- audit log 至少记录哪些远程动作。

### 5. Web MVP 信息架构

结论方向：

- 第一屏应是操作台，不是 landing。
- 信息主轴是 Device -> Project -> Thread / Task，而不是单一聊天窗口。
- 第一版 UI 不做 provider 管理、不做复杂 agent graph。

重点代码：

- `friuns2-codex-mobile/src/api/codexRpcClient.ts`
- `friuns2-codex-mobile/src/server/codexAppServerBridge.ts`
- `friuns2-codex-mobile/src/server/authMiddleware.ts`
- `friuns2-codex-mobile/src/server/httpServer.ts`

需要确认：

- SSE / WebSocket 哪个更适合 Control Plane 给 Web UI 推送。
- pending approval、running turn、terminal output 是否必须出现在同一事件流。
- UI 是否应该直接展示 Codex thread，还是先抽象为 Task timeline。

### 6. iOS 后续扩展

结论方向：

- iOS 不直接持有 Codex/OpenAI secrets。
- iOS 只连接 Control Plane 或 relay，不直接打到本机 app-server。
- iOS 的配对状态和 trusted device 需要从第一版 API contract 预留。

重点代码：

- `Emanuele-web04-remodex/CodexMobile/CodexMobile/Services/CodexService.swift`
- `Emanuele-web04-remodex/CodexMobile/CodexMobile/Services/CodexService+SecureTransport.swift`
- `Emanuele-web04-remodex/CodexMobile/RemodexMenuBar/BridgeControlModels.swift`
- `Emanuele-web04-remodex/CodexMobile/RemodexMenuBar/BridgeControlService.swift`

需要确认：

- iOS client 需要的最小 API：device list、task list、thread read、send follow-up、interrupt、approval response。
- QR payload 需要包含哪些字段，哪些字段不能进日志。
- Swift 类型是否从 OpenAPI / JSON Schema 生成。

## 排除项

以下内容不进入第一版调研主线：

- Codex Desktop 重打包和 Electron 发布流水线。
- provider proxy、OpenRouter、Telegram、Cloudflare tunnel、Termux。
- 多 agent supervisor、tmux orchestration、MCP handoff。
- App Store / TestFlight / APNs 的发布细节。
- UI 主题、截图素材、营销文案。
