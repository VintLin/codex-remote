# 参考项目技术调研 v0.1

## 结论

`project_referenecs` 里的项目值得调研，但不能照搬。它们解决的问题分成三类：

1. 单设备远程控制 Codex。
2. 多 agent / 多 provider 编排。
3. Codex app-server 协议封装和本地 Web UI。

本项目第一版的核心不是上述任一类，而是：

> 自托管多设备 Codex 控制台：在不依赖同一 OpenAI / ChatGPT 账号的前提下，把多台设备上各自的 Codex 对话、状态和任务归属聚合到一个 Web 控制台。

因此第一版应吸收它们的安全边界和协议经验，避免吸收过重的多 agent 编排和桌面重打包复杂度。

## 可直接参考

### Sunwood-ai-labs/codex-remote-control-lab

价值最高。它验证了一个关键边界：

```text
Browser / phone
  -> token-protected Node bridge
    -> ws://127.0.0.1:<port>
      -> Codex app-server
```

可参考点：

- Codex app-server 保持在 `127.0.0.1`。
- 对外只暴露桥接层，不裸露 app-server。
- token URL / token-protected API 是最低可用安全模型。
- `thread/read`、`thread/list`、history sync 需要额外处理，不能假设 Codex Desktop 已打开的窗口会实时同步。
- `probe:ws` 这类协议探针应该成为本项目第一阶段任务。

不应照搬：

- 单 active thread 的产品模型。
- phone-first UI。
- 面向 demo 的 token URL 作为长期身份模型。

### friuns2/codex-mobile

价值高。它验证了 browser-first Codex UI、CLI 包装、provider proxy、移动端适配等方向。

可参考点：

- Express + WebSocket / SSE 桥接 app-server。
- 前端可以通过普通浏览器复刻大部分 Codex App 操作体验。
- `@xterm/xterm` 可用于后续终端输出和交互。
- provider proxy 说明了不要把第三方 provider key 混进 Codex 进程配置之外的地方。

不应照搬：

- 第一版不要做 provider proxy。
- 第一版不要做 Telegram、Cloudflare tunnel、Termux 等扩展入口。
- 第一版不要把“运行 Codex App UI anywhere”作为定位。

### getpaseo/paseo

价值中高，主要用于后续 iOS / desktop / CLI 扩展的 monorepo 参考。

可参考点：

- daemon + clients 的形态适合本项目长期方向。
- protocol / client / server / cli / app 分包边界清楚。
- self-hosted、多 provider、mobile / desktop / web / CLI 是后续形态参考。
- worktree isolation 是未来任务执行安全能力参考。

不应照搬：

- 多 provider / 多 agent 抽象第一版不做。
- Expo / desktop / relay 全量工程复杂度不应进入 v0.1。
- 发布体系和 workspace 数量对当前阶段过重。

### Emanuele-web04/remodex

价值中。它提供了 iOS remote client、QR pairing、trusted reconnect、relay/self-hosting 的产品参考。

可参考点：

- iOS 后续扩展应按“本机 Worker + 配对 / 可信设备”设计，而不是让 iOS 直接持有 Codex/OpenAI secrets。
- QR bootstrap 和 trusted reconnect 是移动端自然接入方式。
- 本地 Codex runtime 留在 Mac / Windows，移动端只做控制端。

不应照搬：

- macOS-only launchd 后台服务不能作为本项目唯一 Worker 安装模式。
- iOS 原生应用不是第一版。

### awslabs/cli-agent-orchestrator

价值中低，主要作为“不要做太早”的参考。

可参考点：

- tmux / terminal session isolation 是未来多 agent 编排可能方向。
- Web UI / CLI / MCP control planes 的分层可作为远期参考。

不应照搬：

- supervisor-worker 多 agent 编排不是本项目 MVP。
- tmux 不是跨 Windows / macOS 的最佳第一版抽象。

### Haleclipse/CodexDesktop-Rebuild

价值低。它更像 Codex Desktop 跨平台重打包。

可参考点：

- Electron 打包脚本和跨平台发布可作为远期 desktop shell 参考。

不应照搬：

- 第一版是 Web，不需要重打包 Codex Desktop。

### openai/codex

价值最高，但定位不同。它是协议和行为的上游事实源。

必须参考：

- `codex app-server generate-ts`
- `codex app-server generate-json-schema`
- app-server `thread/*`、`turn/*`、`model/list`、approval、streaming events
- remote WebSocket auth 和 bounded queue 行为

## 对本项目的设计影响

### 采用的边界

```text
Web UI
  -> Control Plane
    -> Device Worker
      -> localhost Codex app-server
      -> local filesystem / git / terminal
```

强约束：

- Control Plane 不保存 OpenAI / ChatGPT / provider secrets。
- 每台设备使用自己的 Codex auth、API key、model provider 和本地配置。
- Codex app-server 不直接对 LAN /公网暴露。
- Worker 是唯一对外桥接层。
- Worker 必须有 project allowlist 和 audit log。

### 第一版不做

- 不做官方账号 remote control 的替代登录系统。
- 不做多 agent 编排。
- 不做自动任务迁移。
- 不做 OpenCode / MiniMax / Claude Code provider 抽象。
- 不做 iOS App，但架构预留 mobile client。
- 不做 provider proxy。

## 建议后续继续调研的问题

1. app-server schema 在不同 Codex 版本之间的兼容策略。
2. Windows 上 Codex app-server / socket / auth 路径差异。
3. `thread/list` 是否能可靠覆盖 Codex App、CLI、app-server 创建的历史对话。
4. approval server request 的完整 UI contract。
5. Worker 作为系统服务的安装方式：macOS launchd、Windows service、手动 foreground。
